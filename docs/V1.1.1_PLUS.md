## Revisit removing 7-tap demo mode (Fable review #13)

**Context:** The 7-tap gesture on AuthScreen's "Save your progress" title activates demo mode. Fable review #13 flagged this as a Guideline 2.3.1 concealment risk. Kept for v1.1.0 because:
- Kinderwell only supports OAuth (Sign in with Apple / Google) ‚Äî no email/password field for the reviewer credentials Apple typically expects.
- Prior submissions have shipped with 7-tap and been approved.
- Removing it while making 14+ other v1.1.0 changes concentrates risk.

**Decision for later releases:** revisit whether to remove the 7-tap once we have PostHog data on demo-mode activation volume in prod.

**What to monitor first:** After v1.1.0 is live for ~2 weeks, check PostHog for the `authMethod: 'demo'` event count (set by `setAuthMethod('demo')` in the 7-tap flow). Filter by `environment: prod`. Expected: ~1-5 activations/week (Apple reviewers only). If we see meaningfully more, the gesture has been discovered by end users (TikTok?) and we should remove it in the next release before Apple notices.

**When we do remove it:** update Apple Review Information to remove the fallback path, keep only the sandbox-purchase primary path. See `docs/DEMO_MODE.md` and `docs/RELEASE_CHECKLIST.md` Phase 9 for the current dual-path text.

**Code cleanup on removal:**
- `src/screens/onboarding/AuthScreen.tsx` ‚Äî remove `handleTitlePress` and the 7-tap state machine
- `src/store/authStore.ts` ‚Äî remove `setDemoUser`, `isDemoUser` field, all demo-user branches
- `src/hooks/useLessonGate.ts` ‚Äî remove the `isDemoUser` short-circuit
- `src/App.tsx` ‚Äî remove the demo-user branch in `onSubscriptionStatusChange`
- Any other `isDemoUser` reference (grep to find)

Estimated effort when we do this: ~1-2 hours code + 30 min App Store Connect notes update.

## Lesson progress not synced to Supabase

**Problem:** `src/services/lessonProgressService.ts` has 7 functions
querying the `lesson_progress` table in Supabase ‚Äî but **nothing in the
app calls any of them**. The service is 100% dead code, and the
matching `lesson_progress` table in Supabase is dead schema (empty for
every user).

Lesson progress today is stored per-lesson in AsyncStorage via
utility files like `src/utils/sprinklersProgress.ts`,
`src/utils/communicationMistakesProgress.ts`, etc. Each lesson has its
own storage key.

**Consequences:**
- Force-quit + reopen ‚Üí progress survives (AsyncStorage persists) ‚úÖ
- Uninstall + reinstall ‚Üí progress lost ‚ùå
- Signed in on second device ‚Üí no progress there ‚ùå
- User buys new iPhone ‚Üí starts over ‚ùå

**Fix scope (~2-3 hours):**
- Modify each of the ~8 progress-tracking utility files to also write
  to Supabase (upsert into `lesson_progress` keyed on user_id +
  lesson_id).
- Add a hydration step on app launch that pulls the user's Supabase
  progress into AsyncStorage if it's more complete.
- Handle the merge-conflict case (device has progress, so does
  server) with last-write-wins or a smarter merge.

**Not a v1.1.0 concern.** Prod has always worked this way; users on the
same device don't notice. Cross-device is a real feature to build
later.

## Notifications feature not implemented

**Problem:** Onboarding asks users if they want notifications.
Preference saved to `user_profiles.notifications_enabled`. But no code
path anywhere in the app actually schedules or sends any notification.
`expo-notifications` isn't imported for scheduling anywhere.

Every user in prod has been asked about notifications, said yes or no,
and never received any.

**Options:**
- Build the feature (schedule daily/weekly reminders based on
  preference). Not trivial ‚Äî requires push notification server-side
  setup, deep-link handling on tap, permission timing.
- Remove the question from onboarding entirely so we're not
  dishonestly asking about a feature that doesn't exist.

**Not a v1.1.0 concern** ‚Äî the current state (silently ignoring the
preference) doesn't hurt users, it just doesn't help them.

## Alert.alert() error message rework

**Problem:** ~14 different failure modes in the app all use the same
`Alert.alert("[X] Failed", "[Y] could not be completed. Please try
again[ or contact support]")` template. None tell the user:
1. What actually went wrong
2. What to do about it (retry? update app? contact support with
   what info?)
3. How to escalate (support email visible or buried?)

**Fix ideas (v1.1.1):**
- Categorize errors: network / server / provider / user-cancelled /
  auth-expired.
- Inline errors near the button that triggered them, not modal
  alerts (looks less like a crash to the user).
- On repeat failures, offer a "Report this to support" button that
  copies the error object + a correlation ID for our Sentry logs.
- Standardize the copy: "Something went wrong on our end" for server
  errors vs "Check your connection" for network errors.

**Not a v1.1.0 concern** ‚Äî pattern is systemic, needs design pass, not
a per-alert fix.
