# Content errata — Naming our Emotions transcription

Issues found during the faithful transcription of the 24 hand-built Naming
Emotions screens (Sub1..4, Screen1..6) in `src/screens/naming-emotions/` into
`src/lessons/content/namingEmotions.ts`. Text was copied VERBATIM (errors
preserved); each observed problem is logged here rather than silently fixed.

- namingEmotions §1–§4 (all screens): quotation marks and apostrophes are straight ASCII (`"..."`, `Let's`, `doesn't`, `We're`, `It's`, `don't`, `you're`, `there's`, `That's`, `you'll`, `You've`) rather than the curly typographic quotes (`"…"`, `'`) used in some other lessons. The screen-3 headings use a real Unicode horizontal ellipsis glyph `…` (U+2026) in "At the surface, this felt like…", and the callouts use a real em dash `—` (U+2014) and en dash `–` (U+2013, "2–3 emotions"). All transcribed verbatim as-is.
- namingEmotions §1–§4 screen2 / screen4 / screen5 (RESOLVED — engine extended): these are genuinely interactive screens — a free-text multiline `TextInput` ("What happened?", placeholder "Describe the moment briefly. Where were you? Who was involved?"), and emotion-picker screens (a picker button opening the shared `EmotionPicker` modal, with a conditional "Why did you feel {emotion}?" follow-up `TextInput`). Rather than drop the interactivity, two block types were added to the SPEC-09 engine (Phase 3): `textInput` and `emotionPicker` (see `schema.ts` / `BlockRenderer.tsx`). Screen 2 → `textInput`; screens 4 & 5 → `emotionPicker` (reusing the existing global `EmotionPicker` component verbatim). The user's input is EPHEMERAL — held in local state and discarded on completion — exactly as in the hand-built screens (which never persisted it). The `LessonController` gates its Next button (visible-but-disabled) until each required input is satisfied, mirroring the source's `disabled={value.trim().length === 0}` / `disabled={!canContinue}`.
- namingEmotions §1–§4 screen3 (RESOLVED — chip colours supported): the emotion "chip" (`HAPPY`/`SAD`/`MAD`/`BAD`) is a single centred pill with one-off per-sublesson colours — HAPPY: bg `#FFFBEB`, border `#F59E0B`, text `#D97706`; SAD: bg `#EFF6FF`, border `#3B82F6`, text `#2563EB`; MAD: bg `#FEF2F2`, border `#EF4444`, text `#DC2626`; BAD: bg `#F3F4F6`, border `#6B7280`, text `#4B5563`. The `cardList` chip item schema was extended with optional `color` (bg), `borderColor`, and `textColor` fields, so these distinct topic colours ARE now reproduced (single-item `chips` `cardList`).
- namingEmotions §1–§4 screen3 + screen6: the source "messageBox" and "infoBox" are `Colors.primaryBg`-tinted, left-aligned rounded boxes holding multiple body lines. They are transcribed as `callout` `highlight` blocks (each source line a separate callout line). The default `highlight` renderer centres text and uses `Colors.surface`, so the tint colour and left alignment are not byte-identical; text is preserved verbatim.
- namingEmotions §1–§4 screen6: the original completion screens are content screens (headline + tinted info box), NOT the sparse canonical `sectionComplete` card. Per the SPEC-09 controller model (all-`content`, like dissociation.ts), they are transcribed as `content` screens; the controller writes the section id ('1'..'4') to the storage key on each section's last screen, reproducing markSubLessonComplete's `['1','2','3','4']` array.
- namingEmotions §1–§4 screen2/4/5 (behavioural, minor): the hand-built input screens wrap their body in a `KeyboardAvoidingView` (content pinned top via `justifyContent: 'space-between'`). The generic `LessonController` renders content centred inside the shared `LessonContainer`, which is itself a `ScrollView` — so the keyboard cannot clip the field (it scrolls into view), but the exact push-up animation and top-pinned layout differ slightly. Not reproduced at the block level to avoid adding `KeyboardAvoidingView` to the controller for all ~300 screens; the ScrollView is a functional equivalent.
- namingEmotions §4 screen5: the picker button label here is "Select an emotion" (not the "(optional)" variant), and the code comment says "Require 2 emotions minimum (not optional)" while the disable logic (`selectedEmotion && why.trim().length > 0`) still allows advancing after a single second emotion — a minor logic/label inconsistency in the source, noted but not altered. Transcribed with the source's actual heading ("What was another emotion present?") and helper ("Try to identify at least one more.").
