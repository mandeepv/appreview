# v1.1.1 — Onboarding polish sprint

Feedback captured 2026-07-04 from Mandeep testing the currently-live
prod app (v1.0.0 Build 8) on iPhone XR with a fresh Apple ID (never
installed Kinderwell before). This is the "first-time user" perspective
that's easy to lose once you've been developing the app for months.

**Scope**: UX polish + one data-corruption bug. Not touching backend,
paywall, or telemetry — those all shipped in v1.1.0.

**Not shipping until**: v1.1.0 is on the App Store and stable. This
release is for polish, not urgency. Take the time to design it right.

---

## Priority buckets

- **P0**: data corruption or user-blocking. Must fix.
- **P1**: hurts conversion or looks unprofessional. Should fix.
- **P2**: nice-to-have polish.

---

## The list

### 1. Splash flash bug (P1)
For a split second before the splash screen shows up, the logo appears
super-huge across the screen with a green background behind it.

Likely cause: iOS launch screen (`SplashScreen` in expo-splash-screen)
is showing the app icon at wrong dimensions before React Native has
mounted the JS splash screen. Or the JS-side splash is racing the
native one.

**Investigation**:
- Check `app.config.js` splash config for image dimensions
- Check whether we're calling `SplashScreen.preventAutoHideAsync()` /
  `hideAsync()` at the right lifecycle points
- Verify `assets/splash.png` is the right size for iPhone screens

### 2. Splash vs "Get Started" screens look identical (P1)
No sense of progress between them. The user sees the same visual, so
it feels like the app is hung, not that they're moving forward.

**Fix ideas**:
- Add subtle transition (fade, scale, motion)
- Make "Get Started" screen visually distinct from splash — different
  layout, different animation, or a proper welcome hero
- Or: eliminate the redundant screen entirely if there's no reason for
  both

### 3. "Who are you parenting as" screen copy (P1)
"The next few questions help us personalize blah blah" — copy looks
unprofessional and out of place. It's placeholder text or feels like
it.

**Fix**:
- Rewrite copy to sound like the rest of the app
- Or: remove the intro line entirely; the question itself tells the
  story

### 4. "Let's personalize this for you" screen (P0 + P1)
Multiple issues:

**4a. Keyboard auto-loads (P1)**
Loading the keyboard directly the moment the screen opens is aggressive
UX. Best-practice apps let the user tap to focus.

Investigate: is `autoFocus={true}` set on the TextInput? If so, remove
it OR at minimum ensure the Continue button is visible.

**4b. Continue button not visible (P1)**
User can't see the Continue button until they fill in the field and
tap Done on the keyboard. That's confusing — they don't know there's a
next step.

**Fix**: use `KeyboardAvoidingView` properly OR make the Continue
button float above the keyboard OR use an "Enter" key on the keyboard
itself that submits.

**4c. Name field is optional but shouldn't be (P0)**
Name being optional means we're missing data on a lot of users, and
personalization is worse. Also weird UX — why ask if it doesn't
matter?

**Fix**: make Name required. Update the field label. Update onboarding
data validation to require it. Backfill: existing users with null name
are fine, we don't rewrite history.

### 5. Age input is tedious (P1)
The +/- stepper interface for age requires too many taps. Someone
who's 34 has to tap 30-something times.

**Fix ideas** (pick one):
- Text input with numeric keyboard, validate range
- Slider with age buckets ("25-30", "31-35", etc.) if we only need
  buckets not exact age
- Wheel picker (iOS-native UIPickerView style)
- Preset buttons: "Under 25", "25-34", "35-44", "45+" — one tap
- Skip the age question entirely if it's not driving personalization

**Data question**: what are we actually using `age` for? If it's not
feeding personalization or analytics segmentation, we might not need
it at all. Cutting a question improves completion rate.

### 6. "Let's personalize this for your children" — GOOD (no action)
User feedback: this screen works well. Don't regress it.

### 7. Phantom "boy" child bug (P0 — data corruption)
User reported: even when they don't input any child information, the
onboarding_data table gets a "boy" child added. This is data
corruption — we're recording something the user didn't tell us.

**Investigation** (do this first, before designing a fix):
- Check `onboardingStore.children` default value in the Zustand store
- Check `saveUserOnboardingData` in `onboardingService.ts` — does it
  push a placeholder child if the array is empty?
- Check the children screen UI — does it pre-populate a "boy" and
  expect the user to change it?
- Query the prod DB for how many users have exactly one "boy" child
  with no other data → sanity check on how common this is

**Fix**: children array must reflect exactly what the user entered.
Empty = empty. No defaults.

### 8. "What feels hardest right now" — Continue button hidden by scroll (P1)
The Continue button doesn't show up unless you scroll down. "Scroll
for more options" text looks out of place / cheap.

**Fix**:
- Make Continue button sticky at the bottom of the screen
- Remove the "Scroll for more options" hint (better solved by making
  the scroll natural or the button always visible)
- Verify screen behavior on iPhone XR (small screen) — the current
  design likely was tested on a larger simulator

### 9. "How have you been feeling lately" — same scroll problem (P1)
Same issue as #8. Fix should be the same solution (sticky Continue
button), applied consistently across all onboarding screens with lots
of options.

**Consider**: audit ALL onboarding screens for this class of bug and
fix them all in one pass. Consistency > per-screen fixes.

## Cross-cutting themes

Reading the list, three patterns emerge that suggest architectural
fixes rather than per-screen tweaks:

**Theme A: Continue button visibility.**
Multiple screens have Continue buttons that hide behind the keyboard
or below the fold. This is a design system problem — the onboarding
navigation footer should be a standard component that's always
visible, whether keyboard is up or not.

**Theme B: iPhone-small-screen QA gap.**
Screens tested fine on iPhone Pro simulators (bigger screens) but
break on iPhone XR (smaller viewport, older aspect ratio). We need to
add "test on iPhone XR (or an SE-class simulator)" to the release
checklist.

**Theme C: Placeholder copy leaked into prod.**
"blah blah" style copy suggests the team was iterating on copy in
staging and some placeholders leaked to prod. Add copy review as a
pre-ship checklist item.
